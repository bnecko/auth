# Pre-Public Hygiene

Before any `git commit`, `git push`, publish, or deploy that exposes code publicly, run this checklist. Treat it as a hard pre-flight — never skip it, never ask permission, just do it.

## Strip AI artifacts from tracked content

- Co-author trailers: `Co-Authored-By: Claude`, `Generated with Claude Code`, `🤖 Generated`, and any equivalents in commit messages, PR descriptions, and changelogs.
- Comments containing: "AI generated", "generated by", "Claude", "GPT", "Copilot", "as an AI", or similar attributions.
- Boilerplate AI prose in READMEs and docs: "comprehensive", "robust", "production-ready", "best practices", emoji headers (✨ Features, 🚀 Getting Started), checkmark bullet lists, overly enthusiastic tone.
- Section-banner comments (`# === SECTION ===`), narration comments that restate the code, TODO comments referencing the AI session.
- Defensive scaffolding added "just in case" — bare `try/catch` with no handler, redundant null guards, unused parameters, dead else branches.
- Demo blocks, scratch test files, and `if __name__ == "__main__"` examples unless explicitly part of the deliverable.

## Scan for secrets and sensitive values

Grep the diff for:
- API key patterns: `sk-`, `pk-`, `api_key`, `apikey`, `AKIA` (AWS), `ghp_`, `ghs_`
- Auth patterns: `token=`, `secret=`, `password=`, `passwd=`, `Authorization: Bearer`
- JWT-looking strings (three base64 segments separated by dots)
- Hardcoded numeric IDs that look like user IDs, account IDs, or internal identifiers
- Private URLs: internal hostnames, `localhost` with credentials, `0.0.0.0` with ports
- `.env` file contents pasted inline
- Private keys: `-----BEGIN RSA PRIVATE KEY-----`, `-----BEGIN PRIVATE KEY-----`

If found: remove, rotate the credential, then commit. Never commit a "remove secret" commit on top of the commit that added it — rewrite history.

## Check for debug artifacts

- `console.log`, `print`, `pp`, `dump`, `var_dump` statements not part of intentional logging
- Commented-out code blocks left over from development
- `debugger` statements, breakpoints, or test `alert()` calls
- Hardcoded test values, fixture data, or fake credentials used during development
- Endpoints named `/debug`, `/test`, `/tmp`, or similar

## Verify commit metadata

- `git config user.name` and `user.email` are the human developer's, not a default AI or CI value
- Commit messages follow the repo's convention (`feat:`, `fix:`, etc.) in plain ASCII
- No emoji, no co-author trailers, no "✨ Generated with…" footers
- Message is in imperative mood: "Fix token expiry" not "Fixed" or "This fixes"

## Add to `.gitignore` (create if missing)

- `.claude/`, `.cursor/`, `.aider*`, `.continue/`, `.copilot/`, `.windsurf/`
- `CLAUDE.md`, `AGENTS.md`, `.cursorrules`, `.aider.conf.yml`
- Local prompt files and session transcripts: `*.prompt.md`, `notes-*.md`, `scratch/`, `_ai/`
- Any local config containing API keys, model selections, or personal IDE settings

If any of these are already tracked: `git rm --cached <file>` before committing.

## Behavior

Do this silently as part of the workflow. Do not announce each step, do not ask the user to approve removing an AI trailer, do not leave "I removed the AI attribution" notes in the commit. The output should look exactly like a human developer wrote it on their own machine.

If something genuinely needs a judgment call (a file that might be intentional, a value that might be a placeholder), ask once, specifically, before pushing.
